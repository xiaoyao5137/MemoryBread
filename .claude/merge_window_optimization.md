# 情节记忆合并窗口优化

## 问题分析

### 原始问题

**现象**：2026-05-06 05:18 新提炼的情节记忆被合并到 2026-05-05 17:47 的旧记录（相隔 12 小时）

**原因**：
1. 旧的合并逻辑没有时间限制
2. 只要相似度 >= 0.72，就会合并到历史记录
3. 导致今天的工作内容被合并到昨天的记录

### 为什么这不合理？

1. **时间语义丢失**：今天查看订单 ≠ 昨天查看订单
2. **工作上下文不同**：即使是相同的活动，不同时间段的目的可能不同
3. **周报生成混乱**：无法准确统计每天的工作内容
4. **work_item 无法更新**：合并操作只更新 `occurrence_count`，不会更新 work_item 字段

---

## 解决方案

### 实施的优化

**限制合并窗口为 24 小时**

```python
# 2. 获取所有现有知识条目（仅取最近 500 条，且限制在 24 小时内）
merge_window_ms = 24 * 60 * 60 * 1000  # 24 小时
time_filter = ""
if end_time is not None:
    earliest_time = end_time - merge_window_ms
    time_filter = f" AND end_time >= {earliest_time}"
    logger.debug(f"合并窗口: 24小时内 (end_time={end_time}, earliest={earliest_time})")

cursor = db_conn.execute(
    f"SELECT id, overview, entities, start_time, end_time FROM episodic_memories WHERE overview IS NOT NULL{time_filter} ORDER BY created_at DESC LIMIT 500"
)
```

### 优化效果

**优化前**：
- 新记录与所有历史记录（最近 500 条）比较
- 12 小时前的记录也会被合并
- 时间跨度：无限制

**优化后**：
- 新记录只与 24 小时内的记录比较
- 超过 24 小时的记录不会被合并
- 时间跨度：1 天

## 合并逻辑说明

### 完整的合并判断流程

1. **时间窗口过滤**（新增）
   - 只考虑 24 小时内的记录
   - 超过 24 小时的记录直接排除

2. **语义相似度计算**
   - 使用 embedding 计算余弦相似度
   - 阈值：0.72

3. **实体重叠增强**
   - 同名实体出现在两条记录中，相似度 +0.05
   - 最多增强 0.10（2 个实体）

4. **连续片段保护**
   - 15 分钟内的连续片段不会重复计数
   - 避免同一工作被拆分成多条记录

### 为什么选择 24 小时？

1. **符合日常工作习惯**：一天是自然的工作周期
2. **支持周报生成**：按天统计工作内容更准确
3. **平衡去重和精度**：既避免过度合并，又能去除真正的重复

### 其他可选方案

| 窗口大小 | 优点 | 缺点 |
|---------|------|------|
| 12 小时 | 更精确的时间划分 | 可能产生过多重复记录 |
| **24 小时** | **平衡去重和精度** | **推荐** |
| 48 小时 | 更宽松的合并 | 可能合并不相关的工作 |
| 7 天 | 识别周期性工作 | 时间语义丢失严重 |
