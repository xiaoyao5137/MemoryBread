# 基于情节记忆的统一烤面包自动提炼方案

## 1. 文档目标

本文将“知识（knowledge）/ 文档模板（template）/ 操作手册（SOP）”统一收敛到 **与情节记忆相同的定时或闲时提炼链路** 中，在一次 episodic memory 提炼处理中同时完成三类产物的判断、生成、评分、分流与可追溯入库。

本文重点覆盖：
- 数据模型怎么改
- 后端 orchestrator / service / repo 怎么拆
- 前端 Bake 面板怎么接
- 匹配分 prompt + 规则怎么设计

---

## 3. 产品与技术约束（定稿）

这是本方案必须遵守的固定约束：

1. **知识 / 模板 / SOP 不单独设三套不同时间**
   - 统一复用情节记忆现有定时机制或闲时触发机制
   - 在同一次 episode bake 中完成三类提炼

2. **统一评分分流**
   - 高匹配分：自动入库
   - 中匹配分：进入候选库，由用户在 Bake 里决定是否入库
   - 低匹配分：直接丢弃

3. **自动入库内容可人工修订**
   - 用户可在 Bake 中对系统自动提炼产物执行修改、确认、删除

4. **必须保留来源证据链**
   - 任意一个 knowledge/template/SOP 都要能追溯到 memory / capture / knowledge 来源

5. **优先最小侵入复用现有结构**
   - 先复用 `knowledge_entries`、`bake_templates`、现有 bake handlers/service
   - 必要时新增少量关系表与运行日志表

## 4. 总体架构

### 4.1 统一架构图

```text
定时触发 / 闲时触发
        ↓
拉取待处理 episodic memories
        ↓
Episode 聚合（cluster）
        ↓
统一评分 + 统一提炼（一次输出三类结果）
        ↓
┌────────────────────────────────────────────┐
│ knowledge_result / template_result / sop_result │
└────────────────────────────────────────────┘
        ↓
统一分流
  - 高分：自动入库
  - 中分：候选待审
  - 低分：丢弃
        ↓
Bake UI
  - 候选审核
  - 已入库内容编辑
  - 删除错误内容
  - 查看来源证据
```

### 4.2 核心设计原则

1. **一次扫描，多产物输出**
2. **一个触发入口，复用定时/闲时能力**
3. **统一评分，统一阈值，统一状态机**
4. **自动与人工形成闭环**
5. **产物可追溯、可纠偏、可删除**

## 5. 统一调度设计

## 5.1 调度策略

不再为 knowledge/template/SOP 维护三套 cron。

统一改为一个 Bake Job：
- **触发来源 A：与情节记忆相同的定时触发**
- **触发来源 B：系统闲时触发**

两者都调用同一个后端入口：
- `run_unified_bake_pipeline(trigger_reason)`

其中 `trigger_reason` 仅用于日志与监控：
- `scheduled`
- `idle`
- `manual_debug`

## 5.2 为什么这样设计

优点：
- 避免同一批 episodic memories 被扫 3 次
- 三类提炼物共享同一份上下文与证据
- 更适合做统一评分与统一去重
- 易于观察一次运行产出了哪些内容

## 5.3 与现有调度系统的连接方式

### 定时模式
复用 `core-engine/src/scheduler/runner.rs` 当前调度器机制，只新增一个统一任务模板，例如：
- `bake.unified.extract`

### 闲时模式
复用 `ai-sidecar/idle_compute/engine.py` 当前闲时执行能力，只新增一个任务类型，例如：
- `TaskType::BakeExtract`

无论 cron 还是 idle，最终都调用同一条服务链路。

## 6. 统一提炼流水线设计

## 6.1 Stage 0：增量读取

输入不是全量重扫，而是增量：
- 读取 watermark 之后新增或发生变化的 episodic memories
- 若 memory 已被处理过但其源 capture/knowledge 有更新，可进入重算队列

建议引入：
- `bake_watermarks`（新表）
  - `pipeline_name`
  - `last_processed_ts`
  - `updated_at`

如果希望先最小实现，也可临时复用 preference 存储 watermark。

## 6.2 Stage 1：Episode 聚合

把碎片化 memory 聚合成 `episode cluster`。

聚合信号：
- 相近时间窗口
- 相同 app / win_title / URL
- 相同任务标签 / 关键词
- 相同 source_capture_id / source_knowledge_id
- 相同主题 embedding 相似度

产出：
- `episode_cluster_id`
- `source_memory_ids[]`
- `source_capture_ids[]`
- `source_knowledge_ids[]`
- `merged_summary`
- `evidence_bundle`

## 6.3 Stage 2：统一打分

对于每个 episode cluster，一次性输出三类分数：
- `knowledge_match_score`
- `template_match_score`
- `sop_match_score`

每个分数都给出：
- score
- level（high / medium / low）
- reason
- evidence highlights

## 6.4 Stage 3：统一提炼

当 score 达到 medium/high 时，生成对应候选内容：

### Knowledge 输出
- summary
- overview
- details
- entities
- category
- importance

### Template 输出
- name
- category
- applicable_tasks
- structure_sections
- style_phrases
- replacement_rules
- prompt_hint
- linked_knowledge_ids

### SOP 输出
- source_title
- trigger_keywords
- extracted_problem
- steps
- linked_knowledge_ids
- confidence

## 6.5 Stage 4：统一分流

### 高分（自动入库）
直接落正式库，但标记：
- `creation_mode = auto`
- `review_status = auto_created`

### 中分（候选区）
不直接进入正式使用状态，而进入候选待审：
- `review_status = candidate`
- 可在 Bake 中“接受 / 拒绝 / 修改后接受”

### 低分（丢弃）
不在主界面展示，只保留轻量运行日志与计数。

## 7. 评分与阈值设计

## 7.1 阈值建议

第一版建议统一阈值：
- `score >= 0.80` → high → 自动入库
- `0.50 <= score < 0.80` → medium → 候选
- `score < 0.50` → low → 丢弃

后续可放到设置项中调节。

## 7.2 分数来源

建议采用 **规则分 + LLM 分** 的混合方式：

```text
final_score = 规则分 * 0.35 + LLM 分 * 0.65
```

### 规则分信号

#### Knowledge
- 是否包含明确事实/决策/约束/结论
- 是否跨多个 capture 重复出现
- 是否具备跨任务复用价值
- 是否有清晰实体与上下文

#### Template
- 是否存在稳定结构
- 是否同类文档出现过多次
- 是否能抽取固定章节和固定措辞
- 是否带有可复用 prompt hint

#### SOP
- 是否有触发条件
- 是否有顺序步骤
- 步骤是否可执行
- 是否属于重复性问题处理路径

### LLM 分信号
模型对 episode cluster 输出：
- 三类产物是否适合生成
- 置信度
- 主要证据
- 最终生成草案

## 7.3 统一 Prompt 返回格式

建议统一让模型返回 JSON：

```json
{
  "episode_summary": "...",
  "knowledge": {
    "score": 0.88,
    "reason": "包含明确决策与可复用结论",
    "draft": {
      "summary": "...",
      "overview": "...",
      "details": "...",
      "entities": ["..."],
      "category": "...",
      "importance": 4
    }
  },
  "template": {
    "score": 0.76,
    "reason": "结构较稳定但样本数一般",
    "draft": {
      "name": "...",
      "category": "...",
      "applicable_tasks": ["creation"],
      "structure_sections": [],
      "style_phrases": [],
      "replacement_rules": [],
      "prompt_hint": "..."
    }
  },
  "sop": {
    "score": 0.42,
    "reason": "步骤不稳定，不适合抽 SOP",
    "draft": null
  }
}
```

## 8.4 新增表：bake_runs

用于记录每次统一提炼执行情况。

```sql
CREATE TABLE IF NOT EXISTS bake_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  trigger_reason TEXT NOT NULL,
  status TEXT NOT NULL,
  started_at INTEGER NOT NULL,
  completed_at INTEGER,
  processed_episode_count INTEGER NOT NULL DEFAULT 0,
  auto_created_count INTEGER NOT NULL DEFAULT 0,
  candidate_count INTEGER NOT NULL DEFAULT 0,
  discarded_count INTEGER NOT NULL DEFAULT 0,
  knowledge_created_count INTEGER NOT NULL DEFAULT 0,
  template_created_count INTEGER NOT NULL DEFAULT 0,
  sop_created_count INTEGER NOT NULL DEFAULT 0,
  error_message TEXT,
  latency_ms INTEGER
);
```

用途：
- Overview 展示最近烘焙结果
- 排查为什么某次运行没产出
- 观测高/中/低分的比例

---

## 8.5 新增表：bake_artifact_links

用于保存来源关系，支撑证据回溯。

```sql
CREATE TABLE IF NOT EXISTS bake_artifact_links (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  artifact_type TEXT NOT NULL,
  artifact_id TEXT NOT NULL,
  source_type TEXT NOT NULL,
  source_id TEXT NOT NULL,
  relation_kind TEXT NOT NULL,
  created_at INTEGER NOT NULL
);
```

示例：
- artifact_type = `template`
- source_type = `memory`
- relation_kind = `derived_from`

这样可以支持：
- 模板关联了哪些 memory
- SOP 来源于哪些 capture
- knowledge 是基于哪些 episode 聚合出来的

## 8.6 新增表：bake_watermarks

## 9. 后端设计

## 9.1 服务拆分建议

当前 `BakeService` 已承担：
- template CRUD
- memory init / ignore / promote
- sop adopt
- overview 统计

建议在其上新增一个独立编排层，而不是把所有逻辑继续堆在 `BakeService` 里。

### 推荐拆分

#### 1. `BakePipelineService`
职责：统一入口编排
- 拉取待处理 memories
- 做 episode 聚合
- 调统一评分提炼器
- 调分流入库器
- 写 bake_runs / watermark

#### 2. `EpisodeClusterService`
职责：将 episodic memories 聚成 cluster

#### 3. `BakeScoringService`
职责：规则分 + 模型分

#### 4. `BakeArtifactWriter`
职责：把 knowledge/template/sop 分别写入存储层

#### 5. `BakeReviewService`
职责：候选接受 / 拒绝 / 编辑后接受 / 删除

### 保留 `BakeService`
保留现有 API facade，逐步把旧方法迁到新服务里。

---

## 9.2 统一编排入口

建议新增：

```rust
pub fn run_unified_bake_pipeline(&self, trigger_reason: &str) -> Result<BakeRunResult, ApiError>
```

流程：
1. 创建 bake_run 记录
2. 读取 watermark 之后的 memories
3. 聚合 episode cluster
4. 对每个 cluster 统一评分并生成三类 draft
5. 执行高/中/低分分流
6. 写 artifact links
7. 更新 watermark
8. 完成 bake_run

---

## 9.3 存储写入策略

### Knowledge
继续写 `knowledge_entries`，category = `bake_knowledge`

### SOP
继续写 `knowledge_entries`，category = `bake_sop`

### Template
继续写 `bake_templates`

### Memory 候选
继续写 `knowledge_entries`，category = `bake_article`

换句话说，第一版不推翻现有存储策略，只是在写入前面增加统一 orchestrator。

## 9.4 去重与幂等

必须做，否则自动提炼会快速爆库。

### 幂等键建议
```text
episode_cluster_hash + artifact_type + generation_version
```

### 去重策略

#### Knowledge
若 summary + entities + source cluster 高相似：
- 不新建
- 更新 occurrence_count / updated_at / details 中 source ids

#### Template
若 name/category + structure_sections 高相似：
- 不新建重复模板
- 合并 source_memory_ids / linked_knowledge_ids
- 更新 match_score 为更高值

#### SOP
若 trigger_keywords + extracted_problem + steps 高相似：
- 不新建重复 SOP
- 合并 linked knowledge

## 12. Prompt 与规则引擎设计

## 12.1 Prompt 输入

输入给模型的 episode cluster 建议包含：
- episode 摘要
- 标题 / URL / app / window
- 关键 capture 文本片段
- 已存在关联 knowledge
- 打开次数 / 停留时长 / 编辑动作
- 历史标签

## 12.2 Prompt 任务要求

要求模型一次完成：
1. 判断是否适合生成 knowledge
2. 判断是否适合生成 template
3. 判断是否适合生成 SOP
4. 分别输出 score、reason、draft
5. 严格返回 JSON

## 12.3 模型输出约束

必须强调：
- 如果证据不足，就返回低分和 `draft = null`
- 不得补造不存在的步骤或章节
- reason 必须引用输入中的证据模式
- steps 必须是可执行、可复述的操作，不允许空泛口号

## 12.4 规则引擎前置过滤

为了节省 token，建议先规则过滤：

### 直接低分丢弃
- 停留时间过短
- 无编辑动作
- 重复价值极低
- 内容噪声过大
- 明显是临时页面或系统噪音

### 直接高优先送模
- 明确写方案/PRD/周报/流程说明
- 被多次打开和编辑
- 被多条 knowledge 引用
- 同类主题反复出现

## 13. 实施步骤（建议分期）

## Phase 1：统一自动提炼主链路

目标：先跑通“同一次 pipeline 同时产出三类结果”。

改动：
1. 新增 `BakePipelineService`
2. 新增 `bake_runs` / `bake_watermarks`
3. 统一执行入口 `POST /api/bake/run`
4. memory → knowledge/template/sop 统一评分 + 分流
5. 高/中/低分规则落地

交付标准：
- 能跑一次统一提炼
- 能看到自动入库 / 候选 / 丢弃统计
- Overview 可展示最近运行结果

## Phase 2：候选审核与人工修订

改动：
1. 新增 Candidates 页或候选区
2. 补 accept / reject / accept-with-edit 接口
3. 补 knowledge / sop / template 编辑删除能力
4. 补来源证据展示

交付标准：
- 用户可以在 Bake 中审核候选
- 用户可以修改和删除自动提炼内容

## Phase 3：去重、回溯、反馈学习

改动：
1. 新增 `bake_artifact_links`
2. 做相似合并与幂等控制
3. 用户编辑/删除/拒绝反哺评分策略
4. 运行统计与监控增强

交付标准：
- 提炼结果可追溯
- 重复生成明显下降
- 用户反馈能影响后续质量

## 14. 风险与应对

## 14.1 风险：episode 粒度不稳定

问题：memory 太碎或太杂会导致三类产物质量差。

应对：
- 先做 cluster 聚合
- 对太小或太杂的 cluster 降分

## 14.2 风险：模板与 SOP 混淆

问题：某些内容既像模板又像流程。

应对：
- 保留三类独立 score
- 允许同一 episode 同时产出多个候选
- 由用户在候选区最终确认

## 14.3 风险：自动入库污染内容库

问题：高分误判也会直接进正式库。

应对：
- 强制保留证据链
- 提供明显的删除与修订入口
- 首版可将高分阈值调高一些

## 14.4 风险：重复生成

问题：同一 episode 多次触发会不断重复建记录。

应对：
- watermark
- 幂等键
- 相似合并策略

---

## 15. MVP 最终建议

如果要尽快落地，建议 MVP 只做下面这些：

1. 统一 Bake Pipeline 入口
2. 统一高/中/低分分流
3. 高分自动入库，中分候选，低分丢弃
4. 模板/SOP/知识都带来源 evidence
5. Bake 支持编辑、删除、接受、拒绝
6. Overview 展示最近一次 bake run

也就是说，第一版先把“统一自动提炼 + Bake 人工兜底”跑通，再逐步补强学习反馈与高级去重。

---

## 16. 一句话定稿

**把 knowledge / template / SOP 全部统一纳入与 episodic memory 相同的定时或闲时提炼链路，在单次 episode bake pipeline 中同时完成三类产物的评分与生成，高分自动入库，中分进入 Bake 候选区由用户决定是否入库，低分直接丢弃，并允许用户在 Bake 中对系统自动提炼结果进行修改和删除。**
