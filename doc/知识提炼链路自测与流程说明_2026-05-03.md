## 3. 从 capture -> 情节记忆 -> knowledge/template/SOP 的完整流程

## 3.1 Capture 采集层（Core Engine）
1) 事件监听器按周期发送 `CaptureEvent::Periodic`。
2) `CaptureEngine` 处理事件：合并上下文、隐私判断、截图/OCR、落库 `captures`。
3) 对 periodic 截图做视觉去重（dHash + 时间窗），避免重复截图灌库。

关键规则：
- 采集间隔来自 `user_preferences.privacy.capture_interval_sec`，默认 30 秒。
- 系统空闲超过 300 秒（5 分钟）时跳过本轮采集。
- periodic 去重窗口 5 分钟，同场景 dHash 距离 <= 1 视为重复。

## 3.2 情节记忆（episodic_memories）提炼层（AI Sidecar）
1) 后台处理器周期扫描未处理 captures（`timeline_id IS NULL`、非敏感、有文本、过滤自生成窗口）。
2) 先做语义分组（FragmentGrouper），把连续 captures 切成“工作片段”。
3) 片段成熟后调用 `KnowledgeExtractorV2.extract_merged` 做 LLM 提炼。
4) 产出写入 `episodic_memories`，并回填 `captures.timeline_id`。
5) 并行触发向量化写入（capture 向量 + knowledge 向量）。

关键规则：
- 后台扫描频率：90 秒/轮。
- 每轮批量：8 条（默认配置）。
- 分组规则：
  - `HARD_SPLIT_MINUTES=30`：超过 30 分钟强制切片
  - `SOFT_SPLIT_MINUTES=10`：超过 10 分钟需更高相似度
  - `SAME_TASK_THRESHOLD=0.65`
  - `SAME_TASK_THRESHOLD_SOFT=0.72`
  - `DIFF_TASK_THRESHOLD=0.40`
  - `MIN_GROUP_WAIT=3`
- 最后一组收尾条件（_should_finalize_last_group）：
  - idle>=30m，或
  - 组内>=3 且 idle>=10m，或
  - 当前批次不足 batch_size 且 idle>=10m
- 相似知识合并阈值约 `0.72`，连续时间片（含 15 分钟衔接）不重复计数。

## 3.3 统一 bake 提炼层（Core Engine + Sidecar）
1) 当情节记忆有新产出时，background_processor 自动触发 `POST /api/bake/run`。
2) Core 侧做候选筛选（高价值过滤 + watermark 增量过滤）。
3) 对每个候选调用 sidecar `POST /bake/extract`。
4) sidecar 对同一候选并行/串行执行 3 类提炼：
   - bake_knowledge
   - bake_template
   - bake_sop
5) Core 按 accepted/reject + review_status 入库到各表，并更新 run 统计。

关键规则：
- 候选高价值判定（简化）：
  - `importance >= 4` 或 `user_verified=true`，或
  - `occurrence_count >= 2` 且证据强（high/medium）且满足 history/activity/origin 条件。
- 每次 bake run 有 `limit`（默认接口 clamp 1..100）。
- Core 调 sidecar 提炼超时：360 秒。
- 各类别都可 reject；证据不足时不会强行产出。

| 阶段 | 触发频率 | 关键阈值/规则 |
