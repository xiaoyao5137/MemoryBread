# AX 通用方案深度分析

## 🔍 核心发现

### 1. AXStandardWindow 是通用的

**所有 macOS 应用的主窗口都是 `AXStandardWindow`**：
- ✅ VSCode: AXStandardWindow
- ✅ Chrome: AXStandardWindow  
- ✅ Safari: AXStandardWindow
- ✅ Finder: AXStandardWindow

**优先选择 AXStandardWindow 是正确的通用方案**

### 2. 但性能问题依然存在

#### VSCode 的困境

| 窗口类型 | UI 元素数 | `entire contents` 耗时 |
|---------|----------|----------------------|
| AXUnknown | 0 | 0.2秒 |
| **AXStandardWindow** | **1059-5743** | **0.9-4.3秒** |

**问题**：
- VSCode 标准窗口有 1000+ UI 元素
- `entire contents` 需要 1-4 秒（太慢）
- 当前代码会跳过（> 500 元素限制）

#### 当前代码的保护机制

```rust
// 只在 UI ≤ 500 时执行 entire contents
if total_count ≤ 500 then
    // 提取文本
else
    // 跳过，避免卡住
end if
```

**结果**：VSCode 会被跳过，只能提取 focused element（如果有）

## 🎯 结论

### Q1: 优先选用 AXStandardWindow 是否通用？

**✅ 是的**，所有 macOS 应用的主窗口都是 AXStandardWindow

### Q2: 是否大部分 mac 软件都可以用 AX？

**⚠️ 部分正确**：
- ✅ **简单应用**（UI < 500）：可以用 AX，150-250ms
- ❌ **复杂应用**（UI > 500）：AX 太慢（1-4秒），会被跳过

### Q3: 能否全面开启提升效率？

**❌ 不能完全依赖 AX**：

#### 当前方案已经是最优

```
采集流程:
1. 快速检测 AX 支持（50ms，带缓存）
2. 提取 focused element（快速）
3. 检查 UI 元素数
   - ≤ 500 → entire contents（150-250ms）
   - > 500 → 跳过，降级 OCR（1.8秒）
```

#### 为什么不能全面开启 AX？

| 应用类型 | AX 耗时 | OCR 耗时 | 最优方案 |
|---------|---------|---------|---------|
| 简单应用 | 150-250ms | 1.8秒 | **AX** |
| VSCode 等复杂应用 | 1-4秒 | 1.8秒 | **OCR** |

**复杂应用的 AX 比 OCR 还慢！**

## 💡 优化建议

### 方案1: 保持当前策略（推荐）

```rust
if ui_count ≤ 500 {
    // AX 提取（快）
} else {
    // 降级 OCR（慢但稳定）
}
```

**优势**：
- 简单应用用 AX（快）
- 复杂应用用 OCR（避免卡住）
- 已经是最优平衡

### 方案2: 智能采样（不推荐）

对复杂应用只提取部分 UI 元素：

```applescript
-- 只提取前 200 个元素
if total_count > 500 then
    set total_count to 200
end if
```

**问题**：
- 可能遗漏重要内容
- 仍然需要 0.5-1 秒
- 不如 OCR 全面

### 方案3: 异步 AX + OCR（复杂）

同时执行 AX 和 OCR，谁快用谁：

```rust
let (ax_result, ocr_result) = tokio::join!(
    extract_ax(),
    extract_ocr(),
);
```

**问题**：
- 实现复杂
- 双倍资源消耗
- 收益不明显
